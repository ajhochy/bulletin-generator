## Current Situation

The Tailwind CSS + DaisyUI 4 migration removed `src/css/base.css`, `src/css/editor.css`, and `src/css/pages.css`. Some removed rules were aesthetic, but others were structural/runtime dependencies that Tailwind utility classes did not replace:

- page/tab visibility through `.app-page.active`
- full-height `html` / `body` app shell behavior
- editor sidebar / resize handle / preview grid
- preview and print CSS custom properties
- Format page responsive type grid
- generated editor, calendar, staff, serving, and preview selectors

The recovery strategy is a hybrid:

1. Keep Tailwind/DaisyUI and the component-library modernization.
2. Keep valid runtime migrations such as native `<dialog>`, `<progress>`, and DaisyUI toast alerts.
3. Do not restore `base.css`, `editor.css`, or `pages.css` wholesale.
4. Keep `src/css/compat.css` as a post-Tailwind compatibility layer for structural/state CSS.
5. Add minimal fixes only for observed regressions.

## Resolved Known Regressions

### Projects bulk action bar visible with zero selection

The Projects bulk bar was visible even when no projects were selected.

Root cause:

- `index.html` gave `#bulk-bar` an always-on `flex` utility class.
- `src/css/tw-input.css` defined `.bulk-bar { display:none }` and `.bulk-bar.visible { display:flex }`, but generated Tailwind utilities place `.flex { display:flex }` later in the cascade.
- `src/js/projects.js` correctly toggles `.visible`; the CSS/markup combination was wrong.

Resolution:

- Remove the always-on `flex` class from `#bulk-bar`.
- Let `.bulk-bar.visible { display:flex }` control display state.

### Generated paragraph and volunteer editor controls

Runtime spot-checking also showed paragraph break buttons and volunteer role/name controls were generated but missing their structural control styling. Minimal selector rules for those generated controls now live in `compat.css`.
