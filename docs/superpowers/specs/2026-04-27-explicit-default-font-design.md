# Design: Explicit Default Font in Locked Template

**Date:** 2026-04-27
**Status:** Approved

---

## Problem

The locked default template's font family is `system-ui, -apple-system, sans-serif`. This resolves to San Francisco on macOS but to an unpredictable system font on Linux/Docker (the Synology server environment). The sections that use `--font-sans` / `--ui-font` — copyright, calendar, serving today, staff & contact — render with different fonts in the PDF than in the preview, because:

- The **preview** renders in the user's browser (macOS) where system-ui = San Francisco
- The **PDF** is generated by headless Chrome on Linux/Docker where system-ui resolves differently

OOW body text (which uses `--font-serif` = Georgia) is unaffected because Georgia is a reliable cross-platform font.

---

## Solution

Change `DEFAULT_TEMPLATE_CSS_VARS.fontFamily` in `src/js/state.js` from:

```js
fontFamily: 'system-ui, -apple-system, sans-serif',
```

to:

```js
fontFamily: 'Arial, Helvetica, sans-serif',
```

Arial is cross-platform, web-safe, and already the PDF fallback appended by `buildPrintDocHtml()`. Making it the explicit default eliminates the ambiguity at the source.

---

## Scope

### Files changed

| File | Change |
|------|--------|
| `src/js/state.js` | Change `fontFamily` in `DEFAULT_TEMPLATE_CSS_VARS` (1 line) |

### Automatic propagation (no additional changes needed)

- `applyDocTemplate()` reads `DEFAULT_TEMPLATE_CSS_VARS` when no custom template is active — sets `--font-sans` and `--ui-font` in the live preview to Arial automatically
- `buildPrintDocHtml()` reads the same value to write the same CSS vars into the PDF `:root` block — preview and PDF agree

### Minor cosmetic cleanup (in-scope, same PR)

`buildPrintDocHtml()` appends `, Arial, Helvetica, sans-serif` as a fallback suffix to whatever the template font is. After this change, the default stack becomes `Arial, Helvetica, sans-serif, Arial, Helvetica, sans-serif` — redundant but harmless. The suffix should be kept (it still provides a correct fallback for custom Google Fonts) but a code comment clarifying this is worthwhile.

---

## Behavior after the change

| Scenario | Before | After |
|----------|--------|-------|
| Default template, macOS preview | San Francisco (system-ui) | Arial |
| Default template, PDF on Linux | Unpredictable system font | Arial |
| Custom template with Google Font, preview | Custom font | Custom font (unchanged) |
| Custom template with Google Font, PDF | Custom font + Arial fallback | Custom font + Arial fallback (unchanged) |
| "Build from default" new template | Inherits system-ui | Inherits Arial |
| Font picker display for default template | "Default" or blank | "Arial" (already in SYSTEM_TEMPLATE_FONTS) |

---

## What this does NOT fix

The occasional content overflow/cut-off in OOW sections is a separate issue: font metric differences between macOS and Linux Chrome cause text to wrap differently even when the font name is identical. That is a candidate for a follow-up fix using the HTTP endpoint PDF rendering approach (server temporarily hosts HTML at a local URL so headless Chrome loads it via HTTP rather than `file://`, giving it full network access and consistent font resolution).

---

## Non-goals

- No changes to the template editor UI beyond what the font picker already handles
- No changes to the user's ability to build custom templates from the default
- No changes to how custom templates with Google Fonts are rendered
